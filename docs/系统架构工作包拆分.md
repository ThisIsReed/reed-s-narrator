### WP-13 主循环业务闭环强化（阶段A）
**目标范围**
- 解决“模块能力已具备，但真实故事回合尚未闭环”的当前主线短板。
- 将物候更新、知识传播、Agent 裁定、状态结算、持久化审计统一收敛到单一 tick 流程中。
- 将现有 demo 中已能分别展示的亮点，提升为同一条运行链上的真实联动能力，而非分段演示。

**问题背景**
- 当前项目的强项在于模块设计清晰、局部机制完整，但主循环组装层仍偏“骨架化”。
- `phenology/`、`knowledge/`、`agents/`、`orchestrator/`、`persistence/` 已分别具备落地能力，但尚未形成“事件触发 -> 可见知识变化 -> 角色意图 -> 裁定结算 -> 状态落库 -> replay 可验证”的单轮闭环。
- 这会导致项目当前更像“高质量机制原型库”，而不是“可持续推进的故事引擎”。

**优化方案**
1. 在 Orchestrator 中显式分层 tick 流程，固定顺序为：
   `Clock -> Phenology -> EventPool -> Knowledge Update -> Spotlight -> ACTIVE Agent -> PASSIVE Update -> Persistence -> Replay Audit`。
2. 将物候硬约束正式纳入 `NarratorController` 主循环，不再作为独立能力悬置在主流程之外。
3. 将 `KnowledgeAssembler.plan_diffusion()` 对应的传播任务接入真实调度，补齐“事件/事实/认知”在 tick 间推进的执行位置与审计记录。
4. 将 ACTIVE 角色执行从“仅调用注入式 runtime”升级为“统一裁定上下文 + 统一结算回写 + 统一日志落库”的标准回合结构。
5. 为 PASSIVE/DORMANT 角色补齐最小一致性更新规则，避免当前主流程对非 ACTIVE 角色只有分层、缺少充分世界演化语义。
6. 在持久化层补齐闭环所需审计面，保证一个 tick 内的关键中间产物都能被回放与对比，而不只是最终 `WorldState`。
7. 将 demo 升级为“闭环验收场景”，要求一个固定 seed 的样例场景能够完整覆盖物候、隔离、裁定、持久化与回放。

**建议拆分**
- WP-13.1 主循环阶段化重构：
  - 为 `NarratorController` 引入清晰的阶段边界和阶段结果对象，避免跨层逻辑继续堆叠在单一方法中。
- WP-13.2 物候与世界状态联动接线：
  - 将 `apply_phenology()` 接入主循环，并将其 `state_changes/audit_log` 纳入统一 tick 结果。
- WP-13.3 知识传播正式落地：
  - 为传播任务引入待执行队列或等价持久化结构，明确任务创建、成熟、执行、审计的时机。
- WP-13.4 Agent 裁定与结算统一化：
  - 将 CharacterAgent / NarratorJudge / DMAgent / fallback 产物统一纳入标准回合输出模型。
- WP-13.5 闭环验收与 demo 场景：
  - 新增一条固定 seed 的端到端样例，作为开发期与回归期共同使用的验收资产。

**验收标准**
- 固定初始世界与固定 seed 下，连续运行与 checkpoint 恢复后继续运行的最终世界状态完全一致。
- 同一 tick 内，不同角色拿到的知识上下文仍满足授权隔离，且传播延迟行为可被 replay 审计。
- 至少存在一条端到端集成测试，覆盖“物候影响 -> 事件生成 -> 知识变化 -> ACTIVE 角色裁定 -> 状态变更 -> 快照/回放”。
- demo 场景不再只是分段展示，而是能在同一条运行链中观察到上述联动结果。

**关键依赖**
- 直接依赖 WP-05、WP-07、WP-08、WP-09、WP-10。
- 应先于 WP-11、WP-12 执行；否则传播增强与可观测性只会叠加在一个尚未收口的主流程骨架上。

## 6. 2026-03-11 架构补强工作包（新增）

### 背景
- 当前主循环已经具备“物候 -> 事件 -> 知识 -> ACTIVE 行动 -> 持久化”的最小闭环，但仍偏叙事驱动与 ACTIVE 角色驱动。
- 本轮审查确认三类能力尚未成为正式运行链路的一等公民：
  - 通用 `RuleEngine` 仍是独立模块，未进入 `NarratorController.run_tick()` 的正式阶段。
  - `InterruptManager` 未接入主循环，长程任务无法被事件或状态变化正式打断。
  - PASSIVE/DORMANT 角色缺少统一执行器，`status_effects`、`long_action` 等持续态尚未形成稳定世界推进语义。
- 因此，后续修复目标不是再补单点模块，而是把“规则 / 中断 / 被动执行”收敛为世界按 tick 推进的正式内核。

### WP-14 通用规则引擎接线与统一结算阶段（阶段A补强）
**目标范围**
- 在 `NarratorController` 中引入正式的 `world_rules` 阶段，使通用 `RuleEngine` 成为主循环中的统一结算器，而不是仅供单元测试的独立工具。
- 明确 `RuleContext` 的运行时来源，覆盖 tick、seed、事件摘要、角色分层结果、主动行动结果等结算上下文。
- 将“物候专用规则链”与“通用世界规则链”分层组织，避免物候系统继续承担本应属于统一规则内核的职责。

**建议改动**
- 在 `orchestrator/` 中新增规则阶段 helper，例如 `apply_world_rules_stage()`，并为其输出单独的阶段审计。
- 扩展 `NarratorController.__init__()`，显式接收 `RuleEngine` 或等价规则执行器。
- 在主循环中固定执行顺序为：
  `clock -> phenology -> interrupt_scan -> event_pool -> knowledge_update -> spotlight -> active_agent -> passive_execution -> world_rules -> persistence -> replay_audit`
- 约束所有非物候类确定性世界变更，优先通过 `RuleEngine` 归并执行，而不是零散散落在外部 resolver 或 demo runtime 中。

**验收标准**
- 至少存在一条集成测试，显式断言 `world_rules` 阶段出现在 tick 阶段链中。
- 固定 seed 下，通用规则执行的 `state_changes + audit_log` 在 snapshot/checkpoint/replay 中完全可复现。
- 新增一类非物候规则样例并进入主循环，例如状态过期、被动资源消耗或长期行动推进。

**关键依赖**
- 依赖 WP-03、WP-09、WP-13。
- 应先于 WP-15、WP-16 落地，因为中断与被动执行都需要统一结算出口。

---

### WP-15 中断链路与长程任务打断闭环（阶段A补强）
**目标范围**
- 将 `InterruptManager` 正式接入主循环，形成“检测 -> 产出 InterruptSignal -> 规则化结算 -> 写回世界状态 -> 审计持久化”的完整链路。
- 让 `long_action` 不再只是聚光灯可用性的输入，而是可被 tick 推进、可被事件打断、可被 checkpoint 恢复的正式运行态。
- 为中断后的恢复策略建立显式语义，例如取消、暂停、改写 remaining ticks、生成后续事件或状态效果。

**建议改动**
- 在 `NarratorController` 中新增 `interrupt_scan` 或 `interrupt_resolution` 阶段，显式接收 `InterruptManager`。
- 为中断结算补齐规则上下文与状态变更投影方式，避免中断逻辑散落在特例代码里。
- 扩展角色长期行为模型；若当前 `long_action: str | None` 信息不足，升级为带进度/剩余 tick 的结构化模型或等价运行态。
- 将中断结果纳入 tick audit、action log 或独立仓储，保证 replay 时能解释“为什么长期任务在该 tick 被打断”。

**验收标准**
- 至少存在一条集成测试覆盖“角色执行长程任务 -> 事件出现 -> 触发中断 -> 长程任务状态变化 -> replay 后一致”。
- 中断信号顺序、命中原因和状态变更可在 tick 审计中直接查看。
- checkpoint 恢复后继续运行，不会重复触发已处理的中断，也不会丢失未处理的长程任务状态。

**关键依赖**
- 依赖 WP-03、WP-04、WP-14。

### WP-16 被动执行层与持续状态推进语义（阶段A补强）
**目标范围**
- 将 PASSIVE/DORMANT 角色从“只重新打标签”升级为“具备最小世界推进语义”的正式执行层。
- 为 `status_effects`、`long_action`、被动资源消耗、状态到期等持续态建立统一执行器，替代当前默认空实现的 `passive_resolver`。
- 明确 PASSIVE 与 DORMANT 的边界：PASSIVE 至少执行持续态结算与局部进展，DORMANT 只执行必要的低成本维护更新。

**建议改动**
- 在 `orchestrator/` 或 `core/` 新增 `PassiveExecutor` / `PassiveRuleSet` 之类的正式组件，由控制器直接装配，而不是依赖外部 lambda。
- 为角色持续态设计稳定输入输出协议，避免不同测试场景各自实现一套被动更新规则。
- 将被动执行结果同样纳入 `state_changes` 与阶段审计，保证 PASSIVE 角色的世界影响可以被 replay 追踪。
- 为 `status_effects` 至少落地一类正式样例，例如剩余回合递减、效果到期移除、持续性资源扣减。

**验收标准**
- 集成测试中，PASSIVE 角色在无 ACTIVE 行动的 tick 也能推动至少一类世界状态变化。
- DORMANT 角色不会错误地产生高成本行为，但其必要维护状态仍可按规则推进。
- 长程运行测试需要显式断言 PASSIVE 执行结果，而不再只断言 ACTIVE 角色和物候变化。

**关键依赖**
- 依赖 WP-02、WP-03、WP-14。
- 与 WP-15 强相关，可并行设计，但应在统一规则阶段落地后接线。
