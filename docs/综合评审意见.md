# 故事叙述者项目综合评审意见（整合版）

> 文档定位：本文件保留“评审结论、关键风险、优先级和修复建议”。  
> 完整架构设计请查看 [`系统架构设计方案.md`](./系统架构设计方案.md)，工作包与当前待修复项请查看 [`系统架构工作包拆分.md`](./系统架构工作包拆分.md)。

## 1. 结论摘要

当前方案方向正确，核心架构（`Narrator` 中央导演 + 多角色代理 + 物候约束）具备可落地性。  
两份外部评审与现有方案的共识是：**必须把“叙事生成”和“世界结算”彻底分离**，并通过统一时间基线、影响传播与严格动作协议，解决一致性和成本问题。

建议将方案收敛为：

1. **统一底层时间**：引入 `Global Tick`（例如 1 tick = 1 小时），叙事粒度仅是 LLM 调用封装。
2. **三态角色执行**：`ACTIVE`（调用 LLM）、`PASSIVE`（规则更新）、`DORMANT`（仅时间推进）。
3. **事件驱动聚光灯**：由“影响圈层 + 权重”选角，不使用纯随机“灵感一闪”。
4. **硬规则结算**：LLM 只给意图和文案，规则引擎负责状态更新与合法性校验。
5. **信息传播系统**：从“隔离”升级到“隔离 + 传播延迟 + 传闻机制”。

---

## 2. 三方评审的核心共识（你 + 优化意见1 + 优化意见2）

1. `Narrator` 必须是唯一权威调度者（时间、信息、仲裁、存档）。
2. 不能让所有角色每轮都调用 LLM，成本与噪声都会失控。
3. 信息隔离必须做成数据层硬隔离，不能只靠 Prompt 口头约束。
4. 物候必须成为“系统因果引擎”，而不只是背景描写。
5. 需要控制“片段精彩但主线松散”的风险（叙事张力管理）。

---

## 3. 关键风险与修正方案（按优先级）

## P0：必须先解决

1. **时间线撕裂风险**
   - 问题：不同角色使用不同粒度时容易不同步。
   - 修正：采用 `Global Tick`。长任务挂载为 `remaining_ticks`，可被事件中断（Interrupt）。

2. **逻辑瞬移/随机穿模**
   - 问题：纯概率“灵感一闪”会破坏地理与因果。
   - 修正：聚光灯选择基于影响拓扑（地点距离、关系距离、事件波及范围），随机只做微扰。

3. **LLM 幻觉直接污染状态**
   - 问题：角色直接输出自由文本动作不可结算、不可校验。
   - 修正：强制结构化输出（`intent` + `flavor_text`），并用动作白名单和参数校验器拦截非法意图。

4. **物候弱化为装饰**
   - 问题：只写进 Prompt 会被忽略。
   - 修正：物候先进入数值层（资源/疾病/迁徙/冲突概率），再驱动叙事事件池。

## P1：应尽快补齐

1. **休眠角色被动影响不足**
   - 方案：引入 `pending_events` 队列 + 影响传播衰减；唤醒时按优先级注入，不做冗长全文摘要。

2. **信息隔离过“机械”**
   - 方案：加入传闻系统、时延传播、可解释推理渠道（角色可猜测但需依据）。

3. **故事整体节奏漂移**
   - 方案：增加“叙述者意图层/戏剧能量池”，平淡积累、冲突释放。

4. **可调试性不足**
   - 方案：所有随机过程使用 `seed`；保证同一存档可重放。

## P2：中期增强

1. 元叙事监控（重复模式、角色高光覆盖率、死亡率控制）。
2. 长周期物候（多年气候趋势、节庆与文化周期）。

## 4. 收敛后的执行要求

本评审不再重复维护完整架构图、数据结构样例和阶段拆分，这些内容统一以下列文档为准：

1. 目标架构、模块边界、接口设计：
   [`系统架构设计方案.md`](./系统架构设计方案.md)
2. 工作包、依赖顺序、当前待修复能力：
   [`系统架构工作包拆分.md`](./系统架构工作包拆分.md)
3. 已实现内容与测试事实：
   [`changelog.md`](./changelog.md)

本评审文档仅保留对这些设计的“约束性判断”：

1. 主循环必须以 `Global Tick` 为唯一时间基线。
2. 主循环必须区分 `ACTIVE / PASSIVE / DORMANT` 三态，而不是让所有角色同成本运行。
3. 一切世界状态变更都必须经过规则化结算，不允许 LLM 直写状态。
4. 信息隔离必须在数据层落地，并允许传播延迟、传闻与推断机制存在。
5. 长程任务、中断、被动执行必须是正式运行链路，而不是测试或 demo 特例。

## 5. 评审视角下的阶段验收

### 阶段 A
- 跑通“无玩家介入”的最小稳定闭环。
- 建立 `Global Tick`、三态执行、动作白名单、结构化结算、物候硬约束、回放一致性。
- 验收重点：
  - 1000 tick 无时间线撕裂。
  - 非法动作拦截率 100%。
  - 平均每 tick LLM 调用数低于总角色数的 40%。

### 阶段 B
- 在阶段 A 闭环稳定后，再引入传播增强与叙事节奏控制。
- 包括影响传播图、传闻与时延传播、戏剧能量池等。
- 验收重点：
  - 休眠角色唤醒后的信息注入长度可控。
  - 主线事件连续性稳定。
  - 固定 seed 下关键状态可重放一致。

## 9. 最终建议（执行层）

1. 先不要扩展角色数量，先把“时间统一 + 规则结算 + 聚光灯分层”做稳。
2. 把“即时模式”硬性限制在 1-3 轮，超限自动结算回到“日”粒度。
3. 任何会改变世界状态的内容都必须经过规则引擎，不允许 LLM 直写数据库。
4. 优先建设调试工具：事件回放、状态差异对比、seed 重跑。

如果按这个收敛版推进，你的系统会同时满足三件事：  
**可控成本、强一致性、可持续扩展**。

## 10. 2026-03-11 审查补充：当前实现缺口与建议修复包

本轮结合代码审查确认，当前仓库已经跑通了“物候 + 事件 + 知识 + ACTIVE 角色 + 持久化”的最小闭环，但还没有完全达到本评审意见中所要求的“完整持续状态模拟闭环”。

### 已确认缺口

1. **通用规则引擎未进入正式主循环**
   - 当前真正接入 tick 闭环的是物候专用规则链；
   - 通用 `RuleEngine` 仍未成为统一世界结算器。

2. **中断机制未接线**
   - 长程任务目前缺少“被事件或状态变化打断”的正式运行链路；
   - 因而 `Interrupt` 还停留在模块存在、运行时未启用的状态。

3. **被动执行层尚未成型**
   - PASSIVE/DORMANT 角色大多只被分层，不稳定推动世界状态；
   - `status_effects`、`long_action` 等持续态缺少统一执行语义。

### 对应修复建议

建议将后续修复聚焦为三项阶段 A 补强工作：

1. **WP-14 通用规则引擎接线与统一结算阶段**
2. **WP-15 中断链路与长程任务打断闭环**
3. **WP-16 被动执行层与持续状态推进语义**

详细拆分、依赖关系与验收标准已追加到 [`docs/系统架构工作包拆分.md`](./系统架构工作包拆分.md)。
